# 基于 AI 的 GitHub PR 代码评审工具技术设计文档

文档版本：v1.0  
适用阶段：项目立项、技术评审、开发落地  
目标平台：GitHub Pull Request  
核心能力：PR 自动接入、变更总结、风险代码识别、AI Review 建议生成、结构化评审报告  

## 1. 项目概述与需求分析

### 1.1 项目背景

Pull Request 代码评审是现代软件工程中保障代码质量、知识共享和风险控制的重要环节。但在真实企业研发流程中，人工 PR Review 长期存在效率和质量不稳定的问题，尤其在多人协作、复杂业务系统、频繁发布和新人参与评审的场景下，问题更加明显。

本项目拟建设一款以 AI 智能分析为核心的 GitHub PR 代码评审辅助工具。用户指定一个 GitHub PR 后，系统自动完成授权校验、PR 信息读取、代码变更解析、上下文补全、智能风险识别、评审建议生成与结构化报告展示，帮助开发者更快理解 PR、发现潜在问题，并提升评审标准的一致性。

该工具不以“替代人工 reviewer”为目标，而是作为 reviewer 的前置分析层和质量辅助层，帮助团队把更多精力放在业务判断、架构判断和关键风险确认上。

### 1.2 传统 PR 评审痛点

1. 理解成本高  
   Reviewer 需要先阅读标题、描述、commit、diff、相关代码和测试文件，才能判断 PR 的真实意图。对于大 PR、多模块 PR 或跨文件逻辑变更，前置理解成本很高。

2. 漏检风险高  
   人工评审容易遗漏边界条件、空指针、权限校验、事务一致性、并发安全、性能退化、测试缺失等隐性问题。复杂调用链中的风险尤其容易被忽略。

3. 评审标准不统一  
   不同 reviewer 的经验、关注点和时间投入不同，导致相同类型问题在不同 PR 中可能被不同程度地关注，团队质量标准难以稳定执行。

4. 新人评审能力不足  
   新成员通常不熟悉项目架构、团队规范和历史问题模式，参与 review 时难以发现深层问题，也难以给出高质量建议。

5. Review 周期长  
   PR 等待 review 和多轮往返会拖慢交付节奏。团队希望更快获得初步风险判断和修改建议。

6. 低价值评论干扰协作  
   如果自动化工具产生大量风格类、主观类或低置信度评论，会降低开发者信任，反而增加沟通成本。

### 1.3 工具核心价值

1. 提升评审效率  
   自动生成 PR 变更总结，帮助 reviewer 快速理解改动范围、业务意图和风险模块。

2. 提升评审质量  
   对常见问题和高危场景进行规则扫描与 AI 语义分析，辅助发现人工容易漏看的问题。

3. 统一评审标准  
   将团队规范、风险规则、模型提示词和输出等级标准化，使不同 reviewer 获得一致的评审辅助信息。

4. 降低新人门槛  
   为新人提供结构化解释和建议，使其可以更快参与有效 review。

5. 保持人工控制权  
   AI 只生成建议和证据，不自动做最终裁决。高置信度建议可推荐发布，低置信度建议仅在工具内展示。

### 1.4 目标用户

1. 一线开发者  
   在提交 PR 前自检代码质量，提前发现问题。

2. Reviewer / Tech Lead  
   快速理解 PR 变更并聚焦高风险问题。

3. 研发团队负责人  
   通过结构化评审数据观察团队质量趋势、问题类型分布和 review 效率。

4. 平台工程团队  
   将工具集成到 GitHub App、CI/CD、质量门禁或内部研发平台。

### 1.5 核心业务需求

1. GitHub PR 自动接入  
   支持用户输入任意 GitHub 公开或私有 PR 链接，完成授权校验后自动拉取 PR 基础信息、变更文件、行级 diff、提交记录和检查状态。

2. PR 变更智能总结  
   自动提炼本次 PR 的业务目标、模块影响范围、新增逻辑、删除逻辑、依赖变更、配置变更、架构影响和测试覆盖情况。

3. 风险代码智能识别  
   识别语法风险、逻辑漏洞、性能隐患、安全漏洞、代码不规范、冗余代码、空指针风险、边界条件缺失、并发风险、不符合团队规范等问题，并尽可能定位到文件和行号。

4. AI Review 建议生成  
   针对每一处风险输出可落地修复建议，并按“必须修复”“建议优化”“仅供参考”分级。

5. 评审结果结构化输出  
   将变更总结、风险清单、分级建议、优化方案和证据统一组织为标准化报告，支持可视化查看、问题定位跳转和 GitHub 评论回写。

### 1.6 非功能性需求

#### 1.6.1 准确性

- 每条风险结论必须包含可追溯证据，例如 diff 行、上下文片段、规则命中原因或调用关系依据。
- 高风险问题优先保证召回率，低风险建议优先控制误报率。
- 对无法确认的问题必须标注“不确定原因”或降低输出等级。

#### 1.6.2 响应速度

- 小型 PR：目标 30 秒内完成完整分析。
- 中型 PR：目标 60 到 120 秒内完成摘要和核心风险分析。
- 大型 PR：目标先在 30 秒内返回初步摘要，再异步完成深度分析。
- 超大型 PR：采用分片、排队和渐进式结果展示。

#### 1.6.3 稳定性

- GitHub API 调用失败需具备重试、限流处理和错误提示。
- AI 调用失败需支持任务恢复、降级模型和结果缓存。
- 分析任务应具备可追踪状态：等待中、拉取中、分析中、完成、失败、部分完成。

#### 1.6.4 易用性

- 用户只需输入 PR 链接即可触发分析。
- 结果页面必须突出“需要立即关注的问题”，避免信息堆叠。
- 所有建议均需可定位、可解释、可筛选。
- 默认不自动发布 GitHub 评论，必须经用户确认。

## 2. 核心功能详细设计

### 2.1 GitHub PR 自动接入能力

#### 2.1.1 功能目标

用户指定 GitHub PR 链接后，系统自动完成 PR 识别、授权校验、数据拉取、diff 解析和分析任务创建。

#### 2.1.2 输入

- GitHub PR URL。
- 用户 GitHub 授权信息或 GitHub App installation token。
- 可选分析配置：是否深度分析、是否包含历史提交、是否允许回写评论。

#### 2.1.3 输出

- PR 基础信息：仓库、PR 编号、标题、描述、提交人、创建时间、更新时间、源分支、目标分支、base sha、head sha。
- 关联提交记录：commit sha、提交人、提交时间、commit message。
- 变更文件清单：文件路径、文件状态、新增行数、删除行数、patch。
- 行级变更信息：新增行、删除行、上下文行、hunk 范围、可评论位置映射。
- 检查状态：CI、测试、lint、构建、安全扫描结果。

#### 2.1.4 实现逻辑

1. PR URL 解析  
   从 PR 链接中解析 owner、repo、pull number。对非法链接给出明确错误提示。

2. 授权校验  
   公开仓库可在低权限模式下读取公开信息；私有仓库需通过 GitHub App 或用户授权读取。企业场景优先采用 GitHub App，因为其权限更细、审计更清晰、撤权更方便。

3. PR 元信息读取  
   读取 PR 标题、描述、作者、分支、状态、是否 draft、base/head sha。

4. 变更文件读取  
   读取所有 changed files，包括新增、删除、修改、重命名文件。对超过 GitHub API 单页限制的文件列表进行分页读取。

5. diff 解析  
   将 patch 解析为文件、hunk、变更行、上下文行和 GitHub review comment 可定位信息。

6. 任务创建  
   将 PR 快照与 diff hash 保存到任务系统。若相同 head sha 已分析过，则复用缓存结果或只做增量分析。

#### 2.1.5 适用场景

- 开发者提交 PR 后自检。
- Reviewer 接到 review 请求后快速理解变更。
- GitHub App webhook 自动触发评审。
- CI 质量门禁前置检查。

### 2.2 PR 变更智能总结功能

#### 2.2.1 功能目标

自动梳理本次 PR 的核心变更，使 reviewer 在阅读代码前就能理解“这次改了什么、为什么改、影响哪些模块、是否有风险”。

#### 2.2.2 输入

- PR 标题与描述。
- commit messages。
- changed files 列表。
- diff 摘要。
- 关键配置文件变更。
- 测试文件变更。

#### 2.2.3 输出内容

1. 业务变更定位  
   描述 PR 可能对应的业务目标、功能点、缺陷修复或重构目标。

2. 模块变更范围  
   按目录、组件、服务、接口、数据库、配置、测试等维度归类。

3. 新增逻辑  
   说明新增能力、接口、流程、分支逻辑或数据结构。

4. 删除或废弃逻辑  
   说明被移除的行为、兼容性风险和潜在影响。

5. 依赖变更  
   识别 package、lockfile、镜像、构建脚本、环境变量等变化。

6. 架构或设计影响  
   判断是否涉及跨模块依赖、公共 API、数据库 schema、消息协议、权限模型或部署配置。

7. 测试覆盖概览  
   说明是否同步新增或修改测试，是否存在明显测试缺口。

#### 2.2.4 输出规范

摘要必须控制在 reviewer 可快速阅读的长度内，优先使用结构化条目，不输出长篇泛泛解释。摘要中必须区分“确定事实”和“模型推断”。例如，文件变更属于确定事实；业务意图若来自标题和 diff 推断，则需标记为推断。

### 2.3 风险代码智能识别功能

#### 2.3.1 功能目标

在 PR 变更范围内识别潜在缺陷、安全风险、性能问题、可维护性问题和团队规范问题，并尽可能定位到具体文件和行号。

#### 2.3.2 风险类型

1. 语法与类型风险  
   包括类型不匹配、未定义变量、错误导入、接口签名不一致、配置格式错误。

2. 逻辑漏洞  
   包括条件判断错误、错误分支遗漏、状态流转异常、数据覆盖、返回值处理错误。

3. 性能隐患  
   包括循环内重复 IO、N+1 查询、大对象拷贝、无界缓存、阻塞调用、未限制分页。

4. 安全漏洞  
   包括注入风险、鉴权缺失、权限绕过、敏感信息泄露、不安全反序列化、弱加密、路径穿越。

5. 代码规范问题  
   包括团队命名规范、异常处理规范、日志规范、错误码规范、API 响应规范不一致。

6. 冗余与可维护性问题  
   包括重复逻辑、死代码、过度复杂分支、职责混乱、难以测试的实现。

7. 空指针与边界条件风险  
   包括 null/undefined 未处理、空数组、空字符串、越界、非法输入、默认值缺失。

8. 并发与一致性风险  
   包括竞态条件、锁粒度错误、事务边界不完整、幂等性缺失、重试导致重复写。

9. 测试缺口  
   包括核心逻辑无测试、边界条件无测试、回归风险无覆盖、快照或契约测试缺失。

#### 2.3.3 实现逻辑

风险识别采用“规则预筛 + AI 语义校验 + 结果归一化”的三段式流程。

1. 规则预筛  
   通过文件路径、关键字、语言特征、diff 行为、依赖变更、测试变更等信号识别候选风险区域。

2. AI 语义校验  
   将候选风险区域连同必要上下文送入模型，让模型判断问题是否真实存在、影响范围是什么、修复建议是否明确。

3. 结果归一化  
   对模型输出进行去重、分级、证据校验和置信度校准。无法提供证据的问题不进入高优先级列表。

#### 2.3.4 行号定位

行号定位基于 diff hunk 映射实现。系统必须同时保存：

- 新文件行号。
- 旧文件行号。
- diff hunk position。
- GitHub inline comment 所需的定位信息。

对于重命名文件、删除文件、二进制文件、超大文件或 GitHub API 不返回 patch 的场景，系统应降级为文件级问题提示，而不是生成不准确的行级评论。

### 2.4 AI Review 评审建议生成功能

#### 2.4.1 功能目标

针对已识别问题生成清晰、可执行、有证据的 review 建议，并按处理优先级分为三个等级。

#### 2.4.2 建议等级

1. 必须修复  
   适用于可能导致安全漏洞、数据错误、线上故障、兼容性破坏、严重性能退化或测试必然失败的问题。默认推荐 reviewer 优先处理。

2. 建议优化  
   适用于影响可维护性、可读性、性能边界、测试完整性或长期演进的问题。建议作者修复，但不一定阻塞合并。

3. 仅供参考  
   适用于代码风格、命名、轻微抽象改进、非关键可读性建议或低置信度观察。默认不回写 GitHub 评论。

#### 2.4.3 建议内容规范

每条建议至少包含：

- 问题标题。
- 文件路径。
- 行号或文件级位置。
- 问题类型。
- 严重级别。
- 置信度。
- 证据说明。
- 影响说明。
- 修复建议。
- 是否建议发布为 GitHub comment。

#### 2.4.4 正常变更的优化建议

对于未发现明显问题的正常变更，系统仍可给出有限的优化性建议，例如：

- 是否需要补充测试。
- 是否需要更新文档。
- 是否需要增加日志或监控。
- 是否需要考虑灰度、兼容或回滚策略。

此类建议必须克制，不应制造无必要噪音。

### 2.5 评审结果结构化输出

#### 2.5.1 功能目标

将所有分析结果聚合为标准化评审报告，便于 reviewer 快速浏览、筛选、定位和发布评论。

#### 2.5.2 报告结构

1. PR 概览  
   包括标题、作者、分支、变更规模、风险等级、分析耗时、模型版本。

2. 变更总结  
   按业务、模块、依赖、配置、测试、架构影响组织。

3. 风险清单  
   按严重级别和问题类型排序，优先展示必须修复项。

4. 分级建议  
   按“必须修复”“建议优化”“仅供参考”分区。

5. 文件风险视图  
   按文件展示风险数量、风险类型、行级定位。

6. GitHub 发布预览  
   展示即将发布的 summary review 和 inline comments，必须允许用户手动选择。

7. 缺失上下文  
   展示模型认为影响判断的信息缺口，例如无法读取完整文件、缺少业务文档、无法定位调用方。

## 3. 系统整体架构设计

### 3.1 分层架构

系统采用分层架构，核心分为接入层、任务层、分析层、模型层、存储层和展示层。

1. 接入层  
   负责 GitHub 授权、PR URL 解析、GitHub API 调用、webhook 接收和 review 回写。

2. 任务层  
   负责分析任务创建、状态流转、异步调度、失败重试、结果缓存和并发控制。

3. 分析层  
   负责 diff 解析、语言识别、风险预筛、上下文组装、结果去重、分级和报告生成。

4. 模型层  
   负责模型选择、prompt 模板、结构化输出、模型重试、降级和结果校验。

5. 存储层  
   负责保存 PR 快照、分析任务、风险结果、审计日志和用户配置。

6. 展示层  
   负责提供 PR 输入、任务进度、报告展示、问题筛选、定位跳转和发布确认。

### 3.2 模块拆分

#### 3.2.1 GitHub Connector

职责：

- 解析 PR URL。
- 读取 PR 元信息。
- 读取 changed files 和 patch。
- 读取 commit 列表。
- 读取 CI/check 状态。
- 发布 GitHub review。
- 处理 GitHub API 限流、分页、权限错误。

#### 3.2.2 Diff Parser

职责：

- 将 patch 解析为结构化 hunk。
- 维护旧行号、新行号、diff position 映射。
- 识别新增、删除、修改、重命名。
- 支持行级评论定位。

#### 3.2.3 Context Builder

职责：

- 根据变更文件补全上下文。
- 获取 hunk 前后代码片段。
- 识别同文件相关函数。
- 识别 import/export 和依赖关系。
- 匹配测试文件。
- 关联历史提交和已有 review comments。

#### 3.2.4 Risk Engine

职责：

- 执行规则预筛。
- 为文件和 hunk 打风险标签。
- 生成模型分析优先级。
- 合并 AI 结果和规则结果。

#### 3.2.5 AI Orchestrator

职责：

- 按任务类型选择模型。
- 组装模型输入。
- 调用模型生成摘要、风险分析和建议。
- 校验结构化输出。
- 对失败调用做重试和降级。

#### 3.2.6 Report Generator

职责：

- 统一输出格式。
- 进行风险排序。
- 生成可视化报告数据。
- 生成 GitHub review 草稿。

### 3.3 数据流设计

标准数据流如下：

1. 用户提交 PR URL。
2. 系统解析 PR 标识并完成授权校验。
3. GitHub Connector 拉取 PR 元信息、文件变更、提交记录和检查状态。
4. Diff Parser 解析 patch 并生成行级映射。
5. Context Builder 补充项目上下文、文件上下文和历史上下文。
6. Risk Engine 执行规则预筛，输出风险候选区域。
7. AI Orchestrator 先生成 PR 摘要，再对高风险区域执行深度分析。
8. Report Generator 归并结果，生成结构化报告。
9. 用户在页面查看、筛选、确认。
10. 系统按用户选择回写 GitHub review。

### 3.4 整体运行流程

1. 任务创建阶段  
   校验输入、识别仓库、验证权限、创建任务记录。

2. 数据采集阶段  
   拉取 PR 数据、diff、commit、CI 状态和必要上下文。

3. 预处理阶段  
   做语言识别、文件分类、diff 解析、风险初筛和上下文压缩。

4. AI 分析阶段  
   先全局摘要，再局部风险分析，最后生成建议。

5. 结果治理阶段  
   做 JSON 校验、证据校验、重复合并、等级调整和置信度校准。

6. 展示与发布阶段  
   输出可视化报告，并支持用户选择发布 GitHub review。

## 4. 核心技术选型方案及设计思路

### 4.1 技术栈选型

#### 4.1.1 前端

建议采用 React 与 Next.js 构建 Web 控制台。原因是：

- 能同时承载页面渲染和 API 层。
- 对企业内部工具开发效率高。
- 易于集成任务轮询、流式展示和权限控制。

#### 4.1.2 后端

MVP 可使用 Next.js API Routes 或独立 Node.js 服务。企业化后建议拆为独立 API 服务和 worker 服务。

核心要求：

- 支持异步任务。
- 支持 GitHub API 调用和回写。
- 支持模型调用和重试。
- 支持任务状态持久化。

#### 4.1.3 数据库

开发阶段可使用 SQLite，企业部署建议使用 PostgreSQL。

需要持久化的数据包括：

- 用户与授权信息。
- 仓库配置。
- PR 快照。
- 分析任务。
- 风险结果。
- 审计日志。

#### 4.1.4 异步队列

MVP 可使用数据库任务状态轮询。生产环境建议使用 Redis Queue、消息队列或云任务服务，以支持并发控制、失败重试和任务优先级。

#### 4.1.5 GitHub 接入

优先选择 GitHub App。

理由：

- 权限粒度更细。
- 支持组织级安装。
- 适合 webhook 自动触发。
- 审计和撤权更符合企业要求。

个人访问令牌可作为本地开发或单用户模式的备选方案。

### 4.2 AI 模型选型依据

AI 模型需同时满足代码理解、跨文件推理、结构化输出、稳定性和成本控制要求。建议采用多模型分层策略，而不是固定使用单一模型。

#### 4.2.1 通用大模型

优势：

- 具备较强自然语言理解和推理能力。
- 能解释业务意图、风险影响和修复建议。
- 适合生成结构化报告和 review comment。

不足：

- 对超大代码上下文成本较高。
- 对特定语言、框架或团队规范需要上下文补充。
- 如果缺少证据约束，容易产生看似合理但不准确的建议。

#### 4.2.2 代码专用模型

优势：

- 对语法、代码结构和常见缺陷模式更敏感。
- 适合补全、重构、局部修复建议。
- 在特定编程语言上可能有更高性价比。

不足：

- 对业务意图、组织规范和跨系统影响的解释能力不一定稳定。
- 对复杂自然语言报告生成可能不如通用强推理模型。
- 企业可用性、审计、稳定 API 和结构化输出能力需单独验证。

#### 4.2.3 本项目推荐策略

采用“强推理模型 + 轻量快速模型 + 可替换模型接口”的组合。

1. 强推理模型  
   用于复杂风险判断、跨文件上下文分析、安全问题、并发问题、数据一致性问题和最终建议生成。推荐用于高风险文件和高价值分析阶段。

2. 轻量快速模型  
   用于 PR 摘要、文件分类、风险预筛结果解释、低风险建议生成。目标是降低成本和提升速度。

3. 模型兼容接口  
   抽象统一的模型调用接口，使系统未来可接入 OpenAI、Azure OpenAI、企业私有模型或本地模型。

### 4.3 零样本、少样本与微调策略

#### 4.3.1 初期采用零样本和规则增强

初期不建议直接微调。原因是：

- 团队高质量标注数据不足。
- Review 风格和业务规范还在探索。
- 微调不一定能解决上下文缺失问题。

应优先通过结构化 prompt、规则预筛、上下文补全和输出校验提升效果。

#### 4.3.2 中期引入少样本示例

当团队积累一定数量的高质量 review 样例后，可在 prompt 中加入少量示例，用于统一建议风格和严重级别判断。

#### 4.3.3 后期评估微调或偏好学习

当系统积累大量已确认的有效建议、误报样本和团队反馈后，可评估微调、偏好排序或自定义分类模型，用于：

- 风险等级校准。
- 团队规范识别。
- 误报过滤。
- 建议风格统一。

### 4.4 结构化输出选型

模型输出必须采用结构化结果，而不是纯文本段落。每次模型调用都应要求返回可校验字段，包括问题类型、严重级别、置信度、证据、建议、是否适合发布到 GitHub。

结构化输出的优势：

- 前端可直接渲染。
- 后端可做规则校验。
- 便于排序、过滤、审计。
- 便于未来统计分析和模型效果评估。

## 5. 代码上下文获取机制设计

### 5.1 设计目标

代码评审的准确性高度依赖上下文。单独分析一行代码往往会导致误判，因此系统需要构建分层上下文，覆盖 PR 增量、文件局部、跨文件依赖、项目全局约定和历史提交信息。

### 5.2 PR 增量代码拉取

系统通过 GitHub PR API 获取以下信息：

- PR 元信息。
- changed files。
- 每个文件的 patch。
- commit 列表。
- base sha 和 head sha。
- 已有 review comments。
- CI/check 状态。

对于 patch 缺失或过大的文件，系统应记录上下文不足，并尝试通过文件内容 API 获取完整文件，再进行局部截取。

### 5.3 diff 上下文拼接

diff 上下文按以下层级拼接：

1. hunk 内新增和删除行  
   这是模型判断的核心证据。

2. hunk 上下文行  
   用于理解局部条件、变量来源和控制流。

3. 函数或方法级上下文  
   若能识别函数边界，应将完整函数纳入分析。

4. 文件级摘要  
   对超长文件不直接全部输入模型，而是生成文件结构摘要，例如导入、导出、类、函数列表和关键依赖。

5. PR 级摘要  
   包括本 PR 的目标、涉及模块和整体影响。

### 5.4 跨文件上下文识别

跨文件上下文主要用于解决调用关系、类型定义、配置依赖和测试覆盖问题。

识别方式包括：

1. import/export 解析  
   根据导入路径寻找被依赖文件或类型定义。

2. 文件路径规则  
   根据目录结构识别 controller、service、repository、model、test 等层次关系。

3. 同名测试匹配  
   根据文件名和目录约定匹配测试文件，例如源文件与测试文件、组件与快照、接口与契约测试。

4. 公共 API 影响识别  
   对导出函数、公开类、接口类型、路由定义、schema、配置项和数据库 migration 做重点标记。

5. 调用链扩展  
   企业版可通过静态分析、AST、语言服务器或代码索引构建调用图，识别调用方和被调用方。

### 5.5 历史代码联动

历史上下文可帮助判断变更意图和风险是否复发。

可关联的信息包括：

- 当前 PR 的 commit messages。
- 最近修改同一文件或同一函数的 commits。
- 历史 PR 中类似文件的 review comments。
- 历史 bugfix 或 rollback 记录。
- 当前文件曾经出现过的高频问题类型。

MVP 阶段可先关联当前 PR 的 commit 列表和已有 comments。企业增强阶段再引入历史 PR 检索和向量索引。

### 5.6 不同变更场景的上下文策略

1. 单文件小改动  
   直接分析 diff hunk 和函数上下文。

2. 跨文件业务改动  
   增加调用关系、测试覆盖和公共 API 影响分析。

3. 配置文件改动  
   重点检查环境差异、默认值、密钥暴露、部署影响和兼容性。

4. 依赖升级  
   重点检查 lockfile、破坏性变更、构建影响和安全漏洞。

5. 数据库 migration  
   重点检查向后兼容、数据回填、回滚策略、索引和锁表风险。

6. 测试代码改动  
   重点检查测试是否真实覆盖业务风险，是否存在快照滥用、断言过弱或 mock 失真。

## 6. 核心难点解决方案

### 6.1 代码分析准确性

#### 6.1.1 问题

AI 容易在缺少上下文时做过度推断，静态规则又难以理解业务语义。单独依赖任一方式都无法满足企业级 review 准确性要求。

#### 6.1.2 方案

1. 多源证据融合  
   将 diff、上下文、规则命中、历史提交、测试变化和 CI 状态合并作为判断依据。

2. 分语言适配  
   根据语言选择不同规则集。例如 TypeScript 关注类型、异步、undefined；Java 关注空指针、事务、线程安全；Python 关注动态类型、异常处理、依赖版本。

3. 分场景分析  
   区分业务代码、工具类代码、配置文件、测试代码、文档、依赖文件。不同场景使用不同判断标准。

4. 强制证据输出  
   模型必须说明依据来自哪段 diff 或上下文。无证据则不得输出为高置信度问题。

### 6.2 上下文理解

#### 6.2.1 问题

PR 只包含增量代码，但问题往往存在于增量代码与旧代码、配置、调用方、测试之间的关系中。

#### 6.2.2 方案

1. 分层上下文构建  
   按 PR、文件、函数、hunk、依赖、历史六个层次构建上下文。

2. 上下文预算控制  
   对模型输入设置 token 预算，优先保留与风险判断相关的信息，避免无差别输入。

3. 相关性评分  
   为上下文片段打分。与 changed lines 越近、调用关系越直接、风险信号越强，优先级越高。

4. 缺失上下文显式暴露  
   当系统无法获取完整上下文时，报告中必须展示“缺失上下文”，避免模型结论被误认为确定事实。

### 6.3 误报与漏报控制

#### 6.3.1 问题

过多误报会降低用户信任，过多漏报会削弱工具价值。代码评审工具必须在二者之间建立可调节机制。

#### 6.3.2 双层校验策略

第一层：规则过滤  
用于快速识别明确风险和排除明显无害变更。

第二层：AI 语义校验  
用于判断规则命中的风险是否真实存在，以及是否值得向 reviewer 展示。

#### 6.3.3 风险权重分级

风险评分由以下维度组成：

- 严重性：问题一旦发生的影响程度。
- 置信度：当前证据是否足以支持判断。
- 影响面：影响单文件、单模块还是公共接口。
- 可修复性：是否能给出明确修复建议。
- 上下文完整度：是否存在重要上下文缺失。

#### 6.3.4 误报剔除逻辑

以下情况应降低等级或不输出：

- 无明确证据。
- 仅基于命名或风格偏好。
- 已有测试覆盖且风险影响较低。
- 问题不在本次 PR 引入。
- 模型无法解释影响路径。

#### 6.3.5 高危漏洞兜底检测

对安全、数据一致性、权限绕过、注入、敏感信息泄露、命令执行、路径穿越等高危类型，除模型分析外应保留规则兜底。即使模型置信度不高，也应以“需人工确认”的形式展示。

### 6.4 接口响应速度

#### 6.4.1 问题

大 PR 可能包含数百个文件和大量 diff，如果同步完整分析，会导致等待时间过长。

#### 6.4.2 方案

1. 分阶段返回  
   先返回任务创建和基础 PR 信息，再返回摘要，最后返回深度分析结果。

2. 分片并行  
   按文件或 hunk 分片分析，高风险文件优先。

3. 增量分析  
   对相同 head sha 或相同 diff hash 复用已有结果。PR 更新后只分析新增变更。

4. 模型分层  
   使用轻量模型完成摘要和分类，强模型只分析高风险区域。

5. 缓存策略  
   缓存 PR 快照、文件摘要、上下文片段、模型结果和报告结果。

### 6.5 用户使用体验

#### 6.5.1 问题

如果工具输出过长、问题优先级不清或定位不准，用户会很快失去使用意愿。

#### 6.5.2 方案

1. 首屏展示关键结论  
   包括总体风险等级、必须修复数量、变更摘要和最高风险文件。

2. 分级展示  
   默认只展开必须修复和高置信度建议，其余内容折叠。

3. 精准定位  
   每条问题必须可跳转到文件、hunk 或 GitHub 行级位置。

4. 可控发布  
   用户可逐条选择是否发布评论，默认不自动打扰 PR 作者。

5. 低噪音原则  
   仅供参考类建议不默认发送到 GitHub。

## 7. 工具输出规范设计

### 7.1 PR 变更总结规范

PR 变更总结应包含以下字段：

1. 一句话摘要  
   用一句话概括本 PR 的主要目的。

2. 业务变更  
   说明对用户功能、业务流程、接口行为或数据处理的影响。

3. 技术变更  
   说明涉及的模块、组件、服务、数据结构和依赖。

4. 代码规模  
   展示变更文件数、新增行数、删除行数、主要文件类型。

5. 风险概览  
   简述最需要 reviewer 关注的风险点。

6. 测试情况  
   判断是否包含测试变更，以及测试覆盖是否与风险匹配。

7. 模型推断说明  
   对非确定事实进行标记，避免把推断当成事实。

### 7.2 风险代码识别报告规范

每条风险必须包含：

- 风险编号。
- 风险标题。
- 风险类型。
- 文件路径。
- 行号或文件级定位。
- 严重级别。
- 置信度。
- 证据来源。
- 影响范围。
- 触发原因。
- 建议处理方式。
- 是否建议发布 GitHub comment。

风险排序规则：

1. 必须修复优先。
2. 高置信度优先。
3. 安全、数据、权限、兼容性问题优先。
4. 影响公共 API 的问题优先。
5. 同一文件内相似问题合并展示。

### 7.3 AI 评审建议输出规范

评审建议应遵循以下原则：

1. 具体  
   指出问题发生在哪里，而不是泛泛说“这里可能有问题”。

2. 可解释  
   说明为什么这是问题，以及可能造成什么影响。

3. 可执行  
   给出明确修复方向，而不是只指出错误。

4. 克制  
   对低价值风格建议减少输出。

5. 分级  
   必须标注“必须修复”“建议优化”或“仅供参考”。

### 7.4 标准化评审报告规范

报告面向两类使用方式：

1. 工具内可视化查看  
   适合 reviewer 阅读、筛选和定位。

2. GitHub Review 回写  
   适合向 PR 作者反馈关键问题。

GitHub 回写内容必须比工具内报告更克制，只发布高价值、高置信度、可定位的问题。

## 8. 性能与精度优化方案

### 8.1 响应速度优化

1. 分阶段分析  
   快速摘要与深度分析解耦，先让用户获得可用结论。

2. 分文件并行  
   不同文件的风险分析可并行执行，但需控制模型并发和 API 限流。

3. 结果缓存  
   以 repo、PR number、head sha、file path、patch hash 作为缓存关键维度。

4. 优先级调度  
   高风险文件优先分析，例如权限、数据库、配置、公共接口文件。

5. 超大 PR 降级  
   对超大 PR 先输出文件级风险和摘要，再提示用户拆分或等待深度分析。

### 8.2 误报过滤策略

1. 证据不足过滤  
   无法定位到 diff 或上下文证据的问题不得作为高优先级输出。

2. 非本 PR 引入过滤  
   如果问题存在于旧代码且本 PR 未扩大影响，应标注为历史问题或不默认展示。

3. 主观建议降级  
   命名、风格、抽象层次等建议默认进入“仅供参考”。

4. 测试覆盖校验  
   若风险已有明确测试覆盖，可降低严重级别，但不直接删除。

5. 多模型或二次确认  
   对高影响但低置信度问题，可触发二次校验。

### 8.3 漏报补全策略

1. 高危规则兜底  
   安全、权限、注入、敏感信息、事务、并发等问题必须有规则扫描兜底。

2. 文件类型专项策略  
   对 migration、配置、依赖、接口定义、权限模块使用专门规则和 prompt。

3. 测试缺口专项分析  
   分析源代码变更与测试变更是否匹配。

4. 历史问题模式复用  
   后续可从历史 review 中学习团队常见问题，形成规则库。

### 8.4 模型推理优化

1. 上下文压缩  
   对长文件先生成结构摘要，只把相关片段送入深度分析。

2. 任务拆分  
   将摘要、分类、风险识别、建议生成拆成多个子任务，避免单次 prompt 过大。

3. 模型分层调用  
   轻量模型处理低风险任务，强模型处理复杂风险。

4. 输出校验  
   对模型输出做结构校验、枚举值校验、证据校验和行号校验。

5. 失败重试与降级  
   对超时、限流和格式错误做有限重试；重试失败后保留部分结果。

## 9. 可扩展性设计与未来迭代方向

### 9.1 架构扩展性

系统应保持核心能力模块化：

- GitHub Connector 可替换为 GitLab、Gitee、Bitbucket Connector。
- Model Provider 可替换为不同 AI 服务。
- Risk Engine 可加载团队规则和语言规则。
- Context Builder 可从简单 diff 上下文演进为代码索引。
- Report Generator 可输出 Web 报告、GitHub 评论、邮件、IM 通知或质量平台数据。

### 9.2 功能扩展

1. 多代码平台适配  
   支持 GitLab Merge Request、Gitee Pull Request、Bitbucket Pull Request。

2. 自定义编码规范  
   团队可配置命名规范、日志规范、异常处理规范、测试覆盖要求和安全规则。

3. 批量 PR 评审  
   支持按仓库、团队或时间窗口批量分析 PR。

4. 历史评审复盘  
   统计问题类型、误报率、修复率、review 耗时和高风险模块。

5. AI 自动修复建议  
   在人工确认后生成 patch 草案或修复说明，但不自动提交。

6. 质量门禁集成  
   与 CI/CD 集成，对严重问题给出阻断建议。

7. 团队知识库集成  
   读取团队编码规范、架构文档、API 规范和历史事故复盘作为上下文。

### 9.3 模型替换兼容

系统应定义统一模型能力接口，至少抽象以下能力：

- 文本生成。
- 代码理解。
- 结构化输出。
- 长上下文处理。
- 工具调用或检索增强。
- 成本和延迟统计。

这样可以在不重写业务逻辑的前提下切换模型供应商或使用企业私有模型。

### 9.4 高阶能力规划

1. 仓库级代码索引  
   构建函数、类、接口、调用关系、测试覆盖和依赖图。

2. 语义检索增强  
   根据变更自动检索相关历史 PR、事故、文档和规范。

3. Review 偏好学习  
   根据用户采纳、忽略和修改反馈优化后续建议排序。

4. 自动生成测试建议  
   根据风险点生成测试场景清单，辅助开发者补测试。

5. 自动修复 PR 草案  
   对简单问题生成可审查修复 patch，并由开发者确认。

## 10. 落地部署与使用流程

### 10.1 本地部署方案

本地部署适用于开发、演示和单人使用。系统可采用单体应用形式运行，包含 Web 控制台、API 服务、轻量数据库和模型调用配置。

本地部署重点关注：

- 配置 GitHub 访问凭证。
- 配置 AI 模型访问凭证。
- 使用本地数据库保存任务和结果。
- 支持输入 PR URL 后完成分析。

本文档不包含具体部署命令，实际部署步骤应在实施阶段单独编写运维说明。

### 10.2 服务端部署方案

服务端部署适用于团队或企业使用。建议将系统拆分为：

- Web 服务。
- API 服务。
- Worker 服务。
- 数据库。
- 缓存和队列。
- 密钥管理服务。
- 日志与监控系统。

服务端部署重点关注：

- 多用户权限隔离。
- GitHub App 安装管理。
- 任务并发控制。
- 模型调用限流。
- 结果审计和数据保留策略。
- 私有仓库数据安全。

### 10.3 GitHub 授权流程

推荐流程：

1. 企业或团队安装 GitHub App。
2. 管理员选择允许访问的仓库。
3. 系统通过 installation token 访问指定仓库 PR。
4. 用户输入 PR URL 或 webhook 自动触发分析。
5. 回写 review 前再次检查用户权限和仓库配置。

授权原则：

- 默认最小权限。
- 不保存长期有效的高权限 token。
- 所有回写操作需要审计。
- 私有仓库数据不用于非授权训练。

### 10.4 PR 关联使用流程

1. 用户进入工具首页。
2. 输入 GitHub PR URL。
3. 系统校验权限并展示 PR 基础信息。
4. 用户选择分析模式：快速分析或深度分析。
5. 系统显示任务进度。
6. 用户查看结构化报告。
7. 用户筛选需要发布的建议。
8. 系统生成 GitHub review 预览。
9. 用户确认后发布。
10. 系统记录审计日志。

### 10.5 Webhook 自动触发流程

1. GitHub PR opened、synchronize、reopened 等事件触发 webhook。
2. 系统校验签名，确认事件来源可信。
3. 系统根据仓库配置判断是否自动分析。
4. 创建分析任务。
5. 分析完成后可在 GitHub PR 中发布摘要链接，或仅在工具内展示。

### 10.6 GitHub PR 内自动评审落地方案

本项目的生产集成方式以 GitHub App 为主。GitHub App 安装到目标仓库后，将 webhook URL 配置为系统的 `/api/github/webhook`。当 PR 发生 `opened`、`reopened`、`synchronize`、`ready_for_review` 事件时，系统自动执行以下流程：

1. 校验 `x-hub-signature-256`，确保 webhook 请求来自 GitHub。
2. 根据 payload 中的 installation id 换取 GitHub App installation access token。
3. 使用 installation token 拉取 PR 元信息、changed files、commits、checks 和 patch。
4. 执行规则预筛与 AI 结构化分析。
5. 生成 GitHub review body 和可定位 inline comments。
6. 调用 GitHub Pull Request Review API 自动发布评审意见。
7. 在 review body 中写入隐藏标记，标记由 PR URL 和 head sha 组成，用于避免同一 commit 重复发布 review。

自动评审默认采用保守策略：

- Draft PR 默认不自动评审，可通过配置开启。
- 默认最多发布有限数量的 inline comments，避免评论噪音。
- 默认以普通 comment 形式发布，不自动 request changes。
- 行级评论定位失败时降级为 summary review，避免整个评审发布失败。
- 高置信度、可定位、有证据的问题才会进入自动 inline comment。

### 10.7 持久化、异步队列与反馈闭环落地方案

当前 MVP 增加文件型持久化适配器，用于保存任务、分析报告、反馈记录和审计日志。数据默认保存在 `.data` 目录：

- `state.json` 保存 review jobs、reports、feedback。
- `audit.jsonl` 以 append-only 方式保存审计事件。

Webhook 不再同步执行完整 PR 分析，而是先创建持久化任务并进入内置队列，然后立即向 GitHub 返回成功响应。队列后台依次处理任务，执行 GitHub 拉取、AI 分析、报告保存和 review 发布。这样可以降低 webhook 超时概率，也为后续替换成 Redis/BullMQ、云队列或独立 worker 服务保留边界。

审计日志记录以下事件：

- 任务入队、开始、成功、失败。
- 手动分析完成。
- 自动 review 发布。
- PR 评论机器人收到消息与回复。
- 用户对建议标记误报、有效或忽略。

误报反馈闭环支持两种入口：

1. Web 报告中对每条 finding 标记“有效 / 误报 / 忽略”。
2. GitHub PR 评论中使用 `/pr-insight false-positive finding:<finding-id> 原因...`。

这些反馈会被持久化保存，后续可用于规则权重调整、prompt 优化、模型评估和团队级误报率统计。

### 10.8 PR 评论机器人交互方案

系统订阅 GitHub `issue_comment` 事件。由于 GitHub PR 同时也是 issue，用户可以直接在 PR conversation 中与机器人交流。当前支持以下命令：

- `/pr-insight explain finding:<finding-id>`：解释某条 AI Review 建议的证据、影响和修复建议。
- `/pr-insight false-positive finding:<finding-id> 原因...`：标记某条建议为误报。
- `/pr-insight valid finding:<finding-id> 原因...`：标记某条建议有效。
- `/pr-insight ignore finding:<finding-id> 原因...`：标记某条建议忽略。
- `/pr-insight reanalyze`：重新分析当前 PR 并回复最新摘要。

评论机器人使用最近一次持久化报告作为上下文。如果找不到报告，会提示用户先重新分析。机器人回复统一写回 PR conversation，不直接修改代码或自动关闭讨论。

## 11. AI 模型选择思路

### 11.1 选择模型时关注的能力

AI 代码评审不同于普通文本总结，模型需要具备以下能力：

1. 代码语义理解  
   能理解函数、类型、异常、异步、事务、并发和数据流。

2. 长上下文处理  
   能在多个文件、commit 和上下文片段之间建立联系。

3. 推理能力  
   能判断代码变更的影响路径和潜在风险。

4. 结构化输出能力  
   能稳定输出可解析、可校验的结果。

5. 低幻觉倾向  
   能在证据不足时明确表示不确定，而不是强行给结论。

6. 成本和延迟可控  
   能在日常大量 PR 分析场景下保持可接受成本。

### 11.2 通用大模型与代码专用模型对比

通用大模型更适合业务理解、跨文件推理、报告生成和复杂问题解释。代码专用模型更适合局部代码理解、补全、修复和语言特定语法分析。

本工具的核心不只是找语法错误，而是判断 PR 变更的真实风险，因此应以强推理通用模型作为深度分析主力，以代码模型或轻量模型作为预筛、补充和局部优化能力。

### 11.3 推荐模型组合

1. 快速分析模型  
   用于 PR 摘要、文件分类、风险候选归类。目标是速度和成本。

2. 深度分析模型  
   用于高风险文件、跨文件逻辑、安全风险、兼容性风险、并发和事务问题。目标是准确性。

3. 校验模型或规则校验器  
   用于检查输出是否有证据、是否重复、是否适合发布到 GitHub。

### 11.4 模型使用策略

1. 默认零样本  
   通过明确任务说明、结构化 schema 和证据要求实现基础可用。

2. 少样本增强  
   逐步加入团队真实高质量 review 样例，提高输出风格一致性。

3. 反馈闭环  
   记录用户采纳、忽略、修改和删除建议的行为，用于后续排序和策略优化。

4. 谨慎微调  
   只有在具备足够标注数据、稳定规范和明确收益时，才考虑微调。

### 11.5 模型兼容性设计

模型调用层应与业务逻辑解耦。业务层只关心模型能力，不关心具体供应商。模型配置应支持：

- 主模型。
- 快速模型。
- 备用模型。
- 每个仓库的模型策略。
- 每类任务的模型策略。
- 成本上限和超时上限。

## 12. 上下文获取方式设计思路

### 12.1 为什么不能只分析 diff

diff 只能展示“改了什么”，但不能完整解释“为什么这样改”“调用方是否受影响”“旧代码是否已有保护”“测试是否覆盖风险”。如果只看 diff，AI 很容易把正常代码误判为问题，也可能漏掉跨文件风险。

### 12.2 上下文获取分层

1. 增量层  
   PR diff、changed files、commit messages。

2. 局部层  
   hunk 前后代码、所在函数、所在类。

3. 文件层  
   文件结构、导入导出、公共接口、相关测试。

4. 模块层  
   同目录文件、调用方、被调用方、配置依赖。

5. 项目层  
   README、贡献规范、架构文档、编码规范、CI 配置。

6. 历史层  
   历史 PR、历史 comments、bugfix commit、事故复盘。

### 12.3 上下文选择原则

1. 相关性优先  
   与 changed lines 直接相关的上下文优先。

2. 风险优先  
   与安全、数据、权限、公共 API、依赖升级相关的上下文优先。

3. 新增优先  
   本次 PR 新增或修改的逻辑优先。

4. 可证据化优先  
   能支撑结论的上下文优先。

5. 预算约束  
   超出上下文预算时，保留摘要而不是完整内容。

### 12.4 依赖关系处理

依赖关系分为代码依赖、配置依赖、数据依赖和运行时依赖。

1. 代码依赖  
   通过 import/export、类型引用、函数调用和文件路径识别。

2. 配置依赖  
   通过环境变量、配置文件、构建脚本和部署配置识别。

3. 数据依赖  
   通过 schema、migration、ORM model、DTO、API response 识别。

4. 运行时依赖  
   通过任务队列、定时任务、消息 topic、缓存 key 和外部服务调用识别。

### 12.5 不同代码变更场景适配

业务逻辑变更应重点理解状态流转和边界条件。公共接口变更应重点关注兼容性和调用方影响。数据库变更应重点关注迁移风险、索引、回滚和历史数据。配置变更应重点关注环境差异和安全。测试变更应重点关注覆盖质量，而不是只看是否存在测试文件。

## 13. 未来扩展方向设计思路

### 13.1 多代码平台扩展

系统应将代码托管平台能力抽象为统一接口。GitHub 是首个实现，后续可扩展到：

- GitLab Merge Request。
- Gitee Pull Request。
- Bitbucket Pull Request。
- 企业自建代码平台。

统一接口包括：

- 拉取变更。
- 拉取上下文。
- 读取评论。
- 发布 review。
- 接收 webhook。
- 权限校验。

### 13.2 自定义编码规范适配

团队可配置自己的 review policy，包括：

- 命名规范。
- 日志规范。
- 错误处理规范。
- 安全规范。
- 测试覆盖要求。
- API 兼容规则。
- 数据库迁移规则。

这些规则应同时用于规则引擎和 AI prompt，保证自动分析符合团队习惯。

### 13.3 团队评审规则自定义

不同仓库风险不同，系统需支持仓库级、团队级和组织级规则继承。

例如：

- 支付系统对数据一致性和审计日志要求更高。
- 前端系统对可访问性、性能和交互一致性要求更高。
- 基础设施仓库对权限、网络、部署和回滚要求更高。

### 13.4 批量 PR 评审

后续可支持按仓库或团队批量分析未 review PR，输出：

- 高风险 PR 排名。
- 需要优先 review 的 PR。
- 长时间未处理 PR。
- 缺少测试的 PR。
- 重复出现的问题类型。

### 13.5 历史评审复盘

系统可沉淀以下指标：

- 问题类型分布。
- 每个仓库高风险文件。
- AI 建议采纳率。
- 误报率。
- 修复耗时。
- reviewer 参与效率。
- PR 规模与缺陷率关系。

这些数据可用于团队质量治理，而不仅是单次 PR 分析。

### 13.6 AI 自动修复代码

未来可在人工确认后生成修复草案。原则是：

- AI 可以生成 patch 建议。
- 开发者必须确认。
- 不自动提交到主分支。
- 对安全、数据迁移、架构变更类问题只给修复方向，不自动改代码。

### 13.7 评审数据统计分析

长期可建设质量看板，包括：

- 代码风险趋势。
- 高危问题趋势。
- 测试缺口趋势。
- 模型误报趋势。
- 团队规范执行情况。
- 仓库质量健康度。

## 14. 安全、隐私与合规设计

### 14.1 数据最小化

系统只读取完成 PR 分析所需的数据，不主动读取无关仓库内容。模型输入应优先使用相关 diff 和上下文，不无差别上传全仓库代码。

### 14.2 权限最小化

GitHub App 权限应按功能最小化配置。只读分析模式不需要写权限；只有用户启用回写 review 时才需要相关写权限。

### 14.3 敏感信息保护

系统需要识别并保护以下内容：

- token、secret、password。
- 私有证书和密钥。
- 内部域名和账号。
- 业务敏感数据。

对疑似敏感信息，应在报告中提示风险，但在日志和审计中避免明文扩散。

### 14.4 审计

系统必须记录：

- 谁触发了分析。
- 分析了哪个仓库和 PR。
- 使用了哪个模型和策略版本。
- 发布了哪些 GitHub 评论。
- 用户采纳或忽略了哪些建议。

### 14.5 数据保留策略

企业可配置数据保留周期。可选策略包括：

- 仅保存结构化结果，不保存完整 patch。
- 分析完成后删除原始上下文。
- 对私有仓库启用更短保留周期。
- 允许管理员按仓库删除历史数据。

## 15. 质量评估与验收标准

### 15.1 功能验收

1. 能成功分析公开 PR。
2. 能成功分析授权私有 PR。
3. 能正确展示 PR 基础信息和变更文件。
4. 能生成结构化变更总结。
5. 能识别并展示风险清单。
6. 能生成分级 review 建议。
7. 能定位到文件和行号。
8. 能在用户确认后回写 GitHub review。

### 15.2 准确性验收

- 高危问题不得因格式或流程问题被静默丢弃。
- 高置信度问题必须包含证据。
- 随机抽样评估 AI 建议有效率。
- 用户可标记误报，为后续优化提供数据。

### 15.3 性能验收

- 小型 PR 在目标时间内完成完整分析。
- 中型 PR 在目标时间内完成摘要和主要风险分析。
- 大型 PR 支持渐进式结果，不阻塞页面。
- GitHub API 限流时有明确降级与提示。

### 15.4 体验验收

- 用户可在 1 分钟内完成首次 PR 分析触发。
- 报告首屏能看清总体风险。
- 问题可按等级、类型、文件筛选。
- 回写 GitHub 前必须有预览和确认。

## 18. 总结

本工具的核心设计思想是：用工程化流程约束 AI，而不是让 AI 无边界地生成评论。系统通过 GitHub PR 自动接入获取真实变更，通过上下文构建解决单行分析误判，通过规则预筛和 AI 语义分析提升问题发现能力，通过结构化输出和证据校验控制误报，通过分阶段分析和缓存机制保障响应速度，通过人工确认和审计机制满足企业使用要求。

最终系统应成为研发团队的 PR Review 助手：帮助 reviewer 更快理解变更、更稳定发现风险、更规范输出建议，同时保留人工最终判断权。
